# 周沫品牌营销 - Vyor Phase 0 GitHub 项目启动流程

---

## 📝 草稿内容

**标题**: GitHub 项目启动：90% 的团队第一步就错了

**正文**:

做技术项目，你是不是也这样？

接到需求 → 建个仓库 → 开干

结果呢？

3 个月后代码乱成一锅粥，新人加入要花 2 天搞懂结构，协作全靠口头沟通，文档永远跟不上代码...

这不是技术问题，是流程问题。

今天用我们刚启动的 Vyor Phase 0 项目，给你看一套完整的 GitHub 项目启动 checklist。

---

## 🔥 最容易踩的 3 个坑

**坑 1：仓库创建太随意**

- 名字拍脑袋决定（今天叫 vyor，明天改 vyor-mvp）
- 描述写不清楚项目定位
- 权限配置一团糟（要么全公开，要么没人能访问）

**坑 2：目录结构没规划**

- src 里所有文件平铺
- 文档和代码混在一起
- 配置文件到处乱放

**坑 3：协作规范缺失**

- 没有.gitignore，node_modules 都提交
- 没有 LICENSE，开源协议不明确
- 没有.env.example，环境变量靠猜

---

## ✅ 正确的启动流程（7 步走）

### Step 1：仓库创建（5 分钟）

**必须明确的 3 件事**：

1. **命名规范**：统一用小写 + 连字符（vyor-mvp，不是 VyorMVP）
2. **可见性**：初期私有，准备好再公开
3. **基础配置**：
   - ✅ Add README
   - ✅ Add .gitignore（选 Node）
   - ✅ Add License（选 MIT）

**判断标准**：外人看仓库名字和描述，30 秒内能懂这是做什么的。

---

### Step 2：目录结构设计（10 分钟）

**推荐结构**（B 端项目通用）：

```
vyor/
├── docs/          # 文档（PRD、架构、API）
├── src/           # 源代码
│   ├── api/       # API 路由
│   ├── services/  # 业务逻辑
│   ├── database/  # 数据库
│   └── utils/     # 工具函数
├── tests/         # 测试文件
├── config/        # 配置文件
└── scripts/       # 脚本文件
```

**核心原则**：按功能模块分，不是按文件类型分。

---

### Step 3：关键文件配置（15 分钟）

**必须有的 5 个文件**：

1. **package.json** — 项目元信息和依赖
2. **tsconfig.json** — TypeScript 配置
3. **.env.example** — 环境变量模板（不要提交真实密钥！）
4. **README.md** — 项目说明和快速开始
5. **LICENSE** — 开源协议

**避坑提醒**：.env 文件必须加进.gitignore，泄露密钥是重大事故。

---

### Step 4：代码规范约定（10 分钟）

**团队必须对齐的**：

- 分支命名：feature/xxx、bugfix/xxx、release/xxx
- 提交信息：[类型] 描述（例：[feat] 添加用户登录功能）
- 代码风格：Prettier + ESLint 统一配置

**判断标准**：任何成员提交代码，其他人能一眼看懂意图。

---

### Step 5：协作权限配置（5 分钟）

**权限分级**：

- Admin：项目负责人（1-2 人）
- Write：核心开发（能 push 代码）
- Read：产品/测试（只能查看和提 issue）

**不要犯的错**：给所有人 Admin 权限，误删分支无法恢复。

---

### Step 6：项目看板搭建（10 分钟）

**GitHub Projects 必配**：

- Todo（待办）
- In Progress（进行中）
- Review（代码审查）
- Done（已完成）

**作用**：不用开会同步进度，看板就是真相。

---

### Step 7：首次提交验收（5 分钟）

**检查清单**：

- [ ] 仓库已创建（私有）
- [ ] README 填写完整
- [ ] .gitignore 已配置
- [ ] LICENSE 已选择
- [ ] 目录结构已创建
- [ ] 关键文件已提交
- [ ] 团队权限已配置
- [ ] 项目看板已搭建

---

## 🎯 边界与风险

**适用场景**：
- 2 人以上协作项目
- 预计开发周期>2 周
- 需要长期维护的 B 端项目

**不适用场景**：
- 个人实验性项目（简单初始化即可）
- 一次性脚本（不需要完整结构）
- 原型验证（先跑通逻辑再说）
